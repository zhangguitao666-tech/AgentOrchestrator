# AI多智能体调度系统 需求分析

## 1. 分析目标

本文档基于 `需求确认.md` 输出正式需求分析结果，目标是将已确认的产品范围进一步整理为可实施的系统分析结论，覆盖以下内容：

- 需求摘要
- 代码库现状
- 业务模型
- 功能模块划分
- 数据结构设计
- API 与 SSE 接口边界
- 前后端职责划分

- MVP 实施建议

## 2. 需求摘要

本项目是一个本地单机演示版 Web 程序，名称为 `AI多智能体调度系统`。

系统核心流程如下：

1. 用户在主会话中与通用大模型进行多轮对话。
2. 通用大模型在主会话内判断当前问题是否属于某个专业任务。
3. 如果命中专业任务，则先给出简短回答，再向用户展示切换建议卡片。
4. 用户确认后，系统生成主会话摘要，创建一个绑定 Dify 智能体的子会话，并自动跳转。
5. 子会话负责后续专业任务处理，继续多轮对话，但不再继续派生其他智能体。

该系统的本质不是“智能体创建平台”，而是“主会话驱动的多智能体调度入口”。

## 5. 业务目标与产品边界

### 5.1 业务目标

- 提供一个统一聊天入口，承载用户与通用模型的自然对话
- 在主会话中识别专业任务并引导用户切换到合适的 Dify 智能体
- 通过主会话和子会话的关系设计，让调度行为可见、可追踪、可恢复
- 以最小实现成本形成可演示的完整业务闭环

### 5.2 明确纳入范围

- 主会话聊天
- 子会话聊天
- 主会话到子会话的智能体切换
- 切换确认卡片
- 主子会话关系展示
- 本地 JSON 持久化
- REST + SSE 接口
- Markdown 和代码块渲染
- Dify 调用失败重试

### 5.3 明确不纳入范围

- 登录和权限
- 会话删除
- 文件上传
- 智能体管理后台
- 多子会话并行派生
- 子会话继续派生
- 流式中断
- 生成过程中继续发送消息

## 6. 核心业务实体分析

系统的核心业务实体可以收敛为 4 类。

### 6.1 会话 Session

职责：承载主会话或子会话的基础信息、状态和关系。

关键属性建议：

- `sessionId`
- `type`，取值 `main` 或 `child`
- `title`
- `createdAt`
- `updatedAt`
- `status`
- `parentSessionId`
- `childSessionId`
- `linkedAgentId`
- `difyConversationId`
- `sourceMessageId`

说明：

- 主会话通过 `childSessionId` 指向自己的唯一子会话
- 子会话通过 `parentSessionId` 指向所属主会话
- `linkedAgentId` 用于标识当前已关联的 Dify 智能体
- `difyConversationId` 仅对子会话有效，用于继续多轮 Dify 对话

### 6.2 消息 Message

职责：保存会话内的所有可见与系统辅助消息。

消息类型建议：

- `user`
- `assistant`
- `system`
- `switch_card`
- `switch_record`
- `error`

关键属性建议：

- `messageId`
- `sessionId`
- `type`
- `role`
- `content`
- `metadata`
- `createdAt`
- `status`

说明：

- `switch_card` 用于承载建议切换卡片
- `switch_record` 用于承载“已派生子会话”记录
- `error` 用于承载失败提示和重试信息
- `metadata` 用于保存卡片状态、目标 agent、失败上下文等结构化附加信息

### 6.3 智能体 AgentConfig

职责：存储通用模型与 Dify 智能体的可运行配置。

`generalModel` 建议字段：

- `baseUrl`
- `apiKey`
- `model`
- `systemPrompt`

`difyAgents[]` 建议字段：

- `agentId`
- `sessionId`
- `name`
- `description`
- `difyAppId`
- `routingPrompt`
- `apiKey`
- `baseUrl`
- `workflowType`

### 6.4 路由结果 RoutingDecision

职责：表达主会话对当前用户消息的路由判断结果。

建议字段：

- `assistantReply`
- `shouldSwitch`
- `targetAgentId`
- `reason`
- `cardPrompt`

说明：

- 该对象是模型内部结构化结果
- 前端只消费用户可见部分，不直接暴露完整原始结构

## 7. 功能模块划分

### 7.1 前端模块

建议拆分为以下最小模块：

1. `AppShell`
负责页面整体布局，承载左侧会话栏和右侧聊天区。

2. `SessionSidebar`
负责主会话列表、新建会话、当前会话高亮和状态展示。

3. `ChatHeader`
负责展示当前会话类型、当前智能体、来源主会话、状态提示和返回入口。

4. `MessageList`
负责消息列表渲染、滚动、卡片消息、错误消息展示。

5. `MessageComposer`
负责输入框、发送按钮、发送中禁用逻辑。

6. `SwitchSuggestionCard`
负责渲染建议切换卡片、确认切换、继续当前助手、卡片状态显示。

7. `ChildSessionSummaryCard`
负责在主会话顶部显示已关联子会话的概览信息和进入按钮。

8. `MarkdownRenderer`
负责 Markdown 与代码块渲染。

### 7.2 后端模块

建议拆分为以下最小模块：

1. `config service`
负责读取和校验 `agents.json`。

2. `storage service`
负责读写 `sessions.json` 和 `messages.json`。

3. `session service`
负责会话创建、标题更新、主子关系维护。

4. `message service`
负责消息落库、查询、状态更新。

5. `routing service`
负责调用通用模型并解析结构化路由结果。

6. `summary service`
负责在确认切换后生成最近三轮摘要。

7. `dify service`
负责调用 Dify chatflow、管理 `conversation_id`、处理重试。

8. `stream service`
负责组织 SSE 阶段事件并向前端输出。

## 10. 后端接口分析

### 10.1 REST 接口

建议最小 REST 集合如下。

1. `GET /api/sessions`
返回主会话列表。

2. `POST /api/sessions`
创建主会话。

3. `GET /api/sessions/{sessionId}`
返回单个会话详情。

4. `GET /api/sessions/{sessionId}/messages`
返回指定会话消息列表。

5. `POST /api/sessions/{sessionId}/switch-confirm`
确认从主会话切换到目标智能体，并创建子会话。

6. `POST /api/sessions/{sessionId}/switch-reject`
拒绝当前切换建议，更新卡片状态。

7. `GET /api/sessions/{sessionId}/child`
获取当前主会话关联的子会话信息。

8. `POST /api/sessions/{sessionId}/retry/{messageId}`
对子会话失败消息执行重试。

### 10.2 SSE 接口

建议最小 SSE 集合如下。

1. `POST /api/sessions/{sessionId}/stream`
向主会话或子会话发送消息并接收阶段化流式响应。

SSE 事件建议统一为：

- `reply_delta`
- `reply_done`
- `switch_suggestion`
- `status`
- `error`

说明：

- 主会话使用该接口时，后端先调用 routing service
- 子会话使用该接口时，后端直接调用 dify service
- 前端只需要基于会话类型决定当前页语义，不需要维护两套流式协议

## 11. 核心服务设计分析

### 11.1 Routing Service

输入：

- 主会话 `sessionId`
- 当前用户消息
- 相关主会话历史
- `agents.json` 中的可路由智能体描述

输出：

- `assistantReply`
- `shouldSwitch`
- `targetAgentId`
- `reason`
- `cardPrompt`

职责：

- 组织通用模型 Prompt
- 强约束模型输出 JSON
- 校验 `targetAgentId` 是否真实存在
- 将无效目标 agent 处理为“不切换”

### 11.2 Summary Service

输入：

- 主会话最近三轮用户-助手往返

输出：

- 一段供 Dify 内部使用的摘要文本

职责：

- 仅在用户确认切换后触发
- 压缩主会话上下文
- 避免把主会话完整历史直接传给子会话

### 11.3 Dify Service

输入：

- 目标 agent 配置
- 当前用户问题
- 主会话摘要
- 子会话已有 `conversation_id`

输出：

- 阶段化流式消息
- 最新 `conversation_id`

职责：

- 首轮消息创建或续用 Dify 会话
- 保存并更新子会话对应 `conversation_id`
- 将 Dify 返回映射为统一 SSE 事件
- 支持失败后按消息重试

### 11.4 Storage Service

职责：

- 统一读写 JSON 文件
- 提供读失败兜底
- 保证写入顺序一致性
- 提供基础查询能力

建议：

- 采用“读入内存对象后覆盖写回”的最简方式
- 每次写入使用原子替换策略，降低 JSON 损坏概率

## 12. 核心时序分析

### 12.1 主会话普通对话时序

1. 前端向 `POST /api/sessions/{sessionId}/stream` 发送用户消息。
2. 后端写入用户消息。
3. 后端调用通用模型执行结构化路由分析。
4. 后端通过 SSE 输出 `reply_delta` / `reply_done`。
5. 若 `shouldSwitch=false`，流程结束。
6. 若 `shouldSwitch=true`，回答结束后输出 `switch_suggestion`。
7. 前端渲染切换建议卡片。

### 12.2 用户确认切换时序

1. 前端点击 `确认切换`。
2. 后端校验主会话尚无子会话。
3. 后端读取最近三轮消息并调用摘要服务生成摘要。
4. 后端创建子会话。
5. 后端将切换卡片状态改为已确认。
6. 后端在主会话中写入 `switch_record`。
7. 后端返回新子会话信息。
8. 前端自动跳转子会话。
9. 前端立刻调用子会话 `stream` 接口自动代发当前问题。

### 12.3 子会话首轮处理时序

1. 前端向子会话 `stream` 接口发送原始用户问题。
2. 后端将主会话摘要与用户问题一起传给 Dify。
3. Dify 返回流式内容。
4. 后端转为统一 SSE 事件推送给前端。
5. 后端保存 `conversation_id`。
6. 前端展示子会话回答。

### 12.4 子会话失败重试时序

1. 子会话调用 Dify 失败。
2. 后端写入 `error` 消息。
3. 前端展示失败提示与 `重试` 按钮。
4. 用户点击重试。
5. 后端重新发送该条失败用户消息到 Dify。
6. 成功后更新状态并写入新回复。

## 13. 边界条件与风险分析

### 13.1 路由误判风险

风险：

- 通用模型可能误判专业任务
- 可能返回错误 `targetAgentId`

控制策略：

- 路由结果必须结构化
- 后端校验 `targetAgentId`
- 只有用户确认后才真正创建子会话
- 无效 agent 结果回退为普通回答

### 13.2 单子会话限制带来的用户预期问题

风险：

- 一个主会话只能派生一个子会话，后续可能再遇到其他专业问题

控制策略：

- 主会话明确提示“当前主会话已有子会话”
- 提供稳定的 `进入子会话` 入口
- 将该限制明确作为 MVP 边界

### 13.3 JSON 持久化可靠性风险

风险：

- 写入中断可能导致文件损坏
- 并发写入容易出现覆盖问题

控制策略：

- 单机演示模式下串行化写入
- 使用临时文件写入后替换原文件
- 启动时执行校验和空数组兜底

### 13.4 流式事件复杂度风险

风险：

- 主会话和子会话如果协议不一致，会增加前端状态复杂度

控制策略：

- 统一 SSE 事件名和阶段语义
- 前端仅区分当前会话类型，不区分流式协议

### 13.5 Dify 上下文一致性风险

风险：

- 子会话只继承摘要，可能缺失细节

控制策略：

- 摘要仅基于最近三轮，控制信息密度
- 子会话继续允许用户补充问题
- 不在 MVP 阶段做更复杂的长期记忆设计

## 14. 非功能性要求分析

### 14.1 可用性

- 用户必须能看清当前是在主会话还是子会话
- 用户必须能看见切换原因和切换结果
- 用户必须能从主会话返回并再次进入子会话

### 14.2 可恢复性

- 浏览器刷新后应能恢复会话历史
- 子会话应能恢复 `conversation_id`
- JSON 读取异常时应有兜底提示

### 14.3 可维护性

- agent 配置集中于 `agents.json`
- 主子会话共用统一消息结构和 SSE 协议
- 接口尽量最小化，降低前后端耦合复杂度

### 14.4 安全性

- 由于是本地单机演示，暂不做权限体系
- 但 `agents.json` 中包含密钥，后续若转在线环境需迁移到服务端环境变量或密钥管理系统

## 15. MVP 实施优先级建议

建议按以下顺序落地：

1. 先完成 JSON 存储、配置校验、会话基础 REST 接口。
2. 完成主会话页面、主会话列表、基础消息渲染。
3. 接入通用模型主会话流式能力。
4. 实现主会话结构化路由和切换建议卡片。
5. 实现确认切换后创建子会话和主子关系展示。
6. 接入 Dify 子会话流式能力和 `conversation_id` 持久化。
7. 补齐失败重试、刷新恢复、状态提示。

## 16. 结论

该需求已经具备直接进入系统设计和原型开发的清晰度，原因如下：

- 产品边界明确
- 主子会话关系明确
- 智能体切换规则明确
- 前后端技术栈明确
- 数据模型和接口模型可以直接落地
- MVP 限制条件明确，能有效控制实现复杂度

从实现角度看，本项目最关键的不是 UI 复杂度，而是以下三个点：

- 主会话结构化路由结果的稳定性
- 主会话到子会话的关系维护
- 主会话流式与 Dify 流式的统一抽象

只要这三点设计稳定，该系统就可以以较低复杂度完成一个可演示、可扩展的第一版。
