# AI多智能体调度系统 需求确认

## 1. 项目定位

- 项目名称：`AI多智能体调度系统`
- 目标形态：最简 Web 单机演示程序
- 使用方式：用户通过聊天界面与通用大模型多轮对话，系统在主会话中判断是否需要切换到已有的 Dify 智能体
- 当前目标：先完成最小可演示 MVP，不做复杂后台能力和平台化能力

## 2. 总体范围

- 前端采用 `React + Vite`
- 后端采用 `Python + FastAPI`
- 流式通信采用 `SSE`
- 数据采用本地 `JSON` 文件持久化
- 通用模型通过 `OpenAI` 兼容接口接入
- Dify 智能体通过后端直接调用 `Dify API`

## 3. 界面结构

- 页面整体为聊天应用布局
- 左侧为主会话列表
- 右侧为当前会话聊天区
- 左侧提供 `新建会话` 按钮
- 左侧只展示主会话，不直接展示子会话
- 主会话按创建时间倒序展示
- 主会话列表中展示：
- 会话标题
- 当前智能体状态
- 更新时间

## 4. 会话模型

- 系统包含两类会话：主会话、子会话
- 主会话由通用大模型负责
- 子会话由单个 Dify 智能体负责
- 一个主会话最多只允许一个子会话
- 子会话不允许继续派生新的子会话
- 子会话标题格式为：`主会话标题 / 智能体名`
- 新建主会话时，标题默认显示为 `新会话`
- 用户发送第一条消息后，主会话标题使用第一句用户消息自动生成

## 5. 主会话规则

- 主会话支持用户与通用大模型进行多轮对话
- 只有主会话执行智能体切换判断
- 子会话不执行智能体切换判断
- 主会话中的一次模型请求需同时完成：
- 输出给用户的简短初步回答
- 是否建议切换智能体的结构化判断
- 命中时返回目标智能体、切换原因、卡片文案
- 当识别为专业任务时：
- 先输出简短初步回答
- 再插入切换建议卡片
- 如果用户拒绝切换：
- 保留切换卡片
- 卡片状态标记为 `rejected`
- 后续继续由当前通用模型回答

## 6. 智能体切换规则

- Dify 智能体由系统切换，不负责创建
- 智能体先采用手工配置方式维护
- 路由判断完全由通用大模型完成
- 路由结果必须采用固定结构返回
- 一次路由只允许命中一个目标智能体
- 如果模型返回不存在的 `targetAgentId`，系统忽略切换，按普通主会话回答继续处理
- 如果主会话已经存在子会话，则禁止再次切换
- 此时系统仅做普通文本提示：当前主会话已有子会话，可前往子会话继续处理
- 同时提供 `进入子会话` 按钮

## 7. 切换确认交互

- 切换建议通过聊天流内卡片展示
- 卡片至少展示以下内容：
- 建议切换到哪个智能体
- 切换原因
- `确认切换`
- `继续当前助手`
- 点击 `确认切换` 后：
- 生成主会话最近三轮用户-助手往返摘要
- 创建子会话
- 在主会话中写入一条“已派生子会话”的系统记录
- 自动跳转到子会话
- 自动将当前这条用户问题发送到 Dify
- 点击 `继续当前助手` 后：
- 保留卡片
- 卡片状态更新为 `rejected`

## 8. 子会话规则

- 子会话绑定单个已配置的 Dify `chatflow` 智能体
- 子会话创建后自动进入该子会话页面
- 子会话允许继续多轮聊天
- 子会话中不再触发新的路由判断和智能体切换
- 子会话页面顶部需要明确展示：
- 当前智能体名称
- 来源主会话
- 返回主会话入口
- 当前状态提示
- 子会话失败时停留在当前页面
- 子会话失败后允许重试刚才失败的那条用户消息

## 9. 上下文继承规则

- 子会话不继承主会话全部历史
- 子会话只继承主会话最近三轮用户-助手往返的摘要
- 摘要由通用大模型生成
- 摘要生成时机为用户确认切换之后即时生成
- 摘要仅供系统内部调用，不在前端界面展示
- 子会话首轮发送给 Dify 的内容包含：
- 当前用户问题
- 主会话最近三轮摘要

## 10. 主会话与子会话联动

- 主会话详情内需要能查看关联子会话
- 进入子会话的入口需要同时存在于以下位置：
- 主会话顶部固定入口 `查看子会话`
- 主会话消息流中的“已派生子会话”记录
- 如果主会话已关联子会话，主会话顶部展示子会话概览卡
- 概览卡至少展示：
- 当前关联智能体
- 当前状态
- `进入子会话`
- 用户从子会话返回主会话后，仍可再次进入子会话继续聊天
- 浏览器刷新后也需要能够恢复主会话、子会话历史和关系

## 11. 消息与展示规则

- 主会话与子会话的消息气泡样式基本一致
- 主子会话只做轻量视觉区分
- 主会话顶部显示 `通用助手`
- 子会话顶部显示 `Dify 智能体名 + 来源主会话`
- 消息渲染至少支持 Markdown 和代码块
- 不支持附件和文件上传

## 12. 流式输出规则

- 主会话支持流式输出
- 子会话支持流式输出
- 流式传输使用 `SSE`
- SSE 按“消息阶段”推送，而不是按 token 推送
- 主会话与子会话统一事件体系
- 建议使用阶段化事件，例如：
- `reply_delta`
- `reply_done`
- `switch_suggestion`
- `error`
- 首版不支持停止生成
- 首版不支持在生成过程中继续发送下一条消息

## 13. 状态定义

- 子会话最简状态定义为：
- `idle`
- `streaming`
- `error`
- 子会话顶部需要有轻量状态提示
- 主会话中的切换卡片需要保留状态信息，至少支持 `rejected`

## 14. 数据存储

- 本地数据拆分为 3 个文件：
- `sessions.json`
- `messages.json`
- `agents.json`
- `sessions.json` 中主会话与子会话共用同一张会话表
- 会话通过 `type: main | child` 区分
- 子会话通过 `parentSessionId` 关联主会话
- 主会话记录关联的子会话信息和目标智能体信息

## 15. 配置文件要求

- `agents.json` 分为两类配置：
- `generalModel`
- `difyAgents[]`
- 通用模型配置也放在 `agents.json`
- `generalModel` 至少包含：
- `baseUrl`
- `apiKey`
- `model`
- `systemPrompt`
- 每个 Dify agent 至少包含：
- `agentId`
- `sessionId`
- `name`
- `description`
- `difyAppId` 或调用标识
- `routingPrompt`
- `apiKey`
- `baseUrl`
- `workflowType`，固定为 `chatflow`
- 每次调用 Dify 时需要传固定 `user` 标识
- 每个子会话需要保存独立的 `conversation_id`
- 首版不提供 Agent 管理页面，统一通过本地配置文件维护

## 16. 消息结构

- 主子会话统一使用一套消息模型
- 消息类型至少包括：
- `user`
- `assistant`
- `system`
- `switch_card`
- `switch_record`
- `error`
- 切换卡片作为独立消息类型存储在 `messages.json`
- 主会话的结构化路由结果不原样暴露给前端
- 前端只接收和展示用户可见内容

## 17. 路由结构化返回

- 主会话中的路由判断返回格式采用强约束 JSON
- 至少包含以下字段：
- `assistantReply`
- `shouldSwitch`
- `targetAgentId`
- `reason`
- `cardPrompt`
- 该结构用于一次请求内同时完成回复与是否切换判断

## 18. 接口风格约束

- 普通数据读写接口采用 REST 风格
- 消息发送与流式返回采用单独的 `/stream` 类接口

## 19. 容错与启动校验

- 启动时需要执行配置校验
- 至少校验：
- `agents.json` 是否存在
- 通用模型配置是否完整
- 每个 Dify agent 必填字段是否完整
- 当 `sessions.json` 或 `messages.json` 损坏时：
- 系统需要兜底为空数组
- 后端控制台输出错误信息
- 前端页面提示“本地数据读取失败”或等价提示

## 20. 当前明确不做的内容

- 不做登录与权限控制
- 不做删除会话
- 不做附件上传
- 不做 Agent 管理页面
- 不做多子会话派生
- 不做子会话继续派生
- 不做流式中断
- 不做生成过程中继续发送消息

## 21. MVP 目标总结

- 用户在主会话中与通用模型正常聊天
- 系统能识别专业任务并建议切换到已有 Dify 智能体
- 用户确认后，系统能基于摘要创建子会话并自动进入该智能体会话
- 主子会话关系清晰、状态可见、历史可恢复
- 整体具备本地演示所需的最小闭环能力
